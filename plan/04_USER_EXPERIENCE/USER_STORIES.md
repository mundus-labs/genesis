# MUNDUS LAB — User Stories

User Stories beschreiben die typischen Nutzungsszenarien von MUNDUS.  
Sie zeigen, wie unterschiedliche Nutzergruppen mit dem System arbeiten,  
welche Schritte sie durchlaufen und welche Ziele sie verfolgen.  
Die Stories sind bewusst einfach, realistisch und menschlich formuliert.

---

## 1. User Story: Der Patent-Analyst

**Als technisch orientierter Nutzer**  
möchte ich ein Patent hochladen  
um es in klare, verständliche Wissensbausteine zerlegt zu bekommen  
damit ich die Struktur des Patents schneller verstehen kann  
und beurteilen kann, welche Mechanismen und Funktionen darin enthalten sind.

**Flow:**
1. Patent hochladen  
2. Roh-Steine betrachten  
3. Steine korrigieren  
4. Beziehungen prüfen  
5. Graphansicht öffnen  
6. Mechanismen nachvollziehen  

---

## 2. User Story: Der Erfinder

**Als Erfinder**  
möchte ich ein technisches Problem analysieren  
um herauszufinden, welche vorhandenen Mechanismen dafür geeignet sind  
und welche Kombinationen neue Lösungen ermöglichen.

**Flow:**
1. Problem definieren  
2. relevante Steine suchen  
3. Mechanismen vergleichen  
4. kombinierbare Steine testen  
5. Sandbox-Vorschläge prüfen  
6. eigene Konstruktion speichern  

---

## 3. User Story: Der Schüler / Lernende

**Als Lernender**  
möchte ich technische Konzepte in kleinen logischen Einheiten verstehen  
um Zusammenhänge zu begreifen  
und mein Wissen aufzubauen, ohne von zu viel Komplexität erschlagen zu werden.

**Flow:**
1. Themengebiet auswählen  
2. Graph erkunden  
3. Steine anklicken  
4. Zusammenhänge sehen  
5. Quiz- oder Check-Fragen in der Werkstatt üben  

(Optional, späteres Feature)

---

## 4. User Story: Der Ingenieur

**Als Ingenieur**  
möchte ich Systeme zerlegen  
um zu verstehen, wie sie funktionieren  
welche Prinzipien sie nutzen  
und wie man sie optimieren kann.

**Flow:**
1. System eingeben oder aus Liste auswählen  
2. Werkstatt öffnet Funktionsketten  
3. Sandbox prüft Optimierungen  
4. Mechanismen vergleichen  
5. neue Version des Systems speichern  

---

## 5. User Story: Der Forscher

**Als Forscher**  
möchte ich technische Alternativen untersuchen  
um herauszufinden, ob es bessere, effizientere oder robustere Konzepte gibt.

**Flow:**
1. Ausgangsmechanismus auswählen  
2. alternative_zu Beziehungen anzeigen  
3. Sandbox schlägt Varianten vor  
4. Vergleichstabelle öffnen  
5. Ergebnisse exportieren  

---

## 6. User Story: Der Community-Mitwirkende

**Als Nutzer, der zum System beitragen möchte**  
möchte ich Roh-Steine korrigieren und verbessern  
um MUNDUS präziser und nützlicher zu machen  
und damit andere bessere Ergebnisse erhalten.

**Flow:**
1. Roh-Stein öffnen  
2. Typ korrigieren  
3. Summary verbessern  
4. Beziehungen ergänzen  
5. Stein validieren  
6. MUNDUS wächst  

---

## 7. User Story: Der Konstrukteur

**Als Konstrukteur**  
möchte ich Komponenten zu einem System zusammenbauen  
um zu testen, ob die Kombination gültig ist  
und ob es alternative Lösungen gibt.

**Flow:**
1. Steine auswählen  
2. Kombination durchführen  
3. Sandbox prüfen lassen  
4. Konflikte beheben  
5. Optimierungen einbauen  
6. System finalisieren  

---

## 8. User Story: Der Analyst von Fehlersituationen

**Als Fehleranalyst**  
möchte ich erkennen, warum ein System nicht funktioniert  
um die fehlerverursachenden Mechanismen zu finden  
und Alternativen testen zu können.

**Flow:**
1. System in Werkstatt laden  
2. Sandbox-Fehleranalyse ausführen  
3. Constraint-Verletzungen prüfen  
4. Dependency-Lücken schließen  
5. Alternativen testen  

---

## 9. User Story: Der Explorer

**Als neugieriger Nutzer**  
möchte ich den Wissensgraph erkunden  
um zufällig neue technische Zusammenhänge zu entdecken  
und zu sehen, wie Mechanismen miteinander verbunden sind.

**Flow:**
1. Graph öffnen  
2. Cluster anzeigen  
3. Knoten anklicken  
4. Funktionspfade verfolgen  
5. Themengebiete wechseln  

---

## 10. User Story: Der Optimierer

**Als technisch erfahrener Nutzer**  
möchte ich bestehende Systeme optimieren  
um Variationen zu testen  
die Kosten, Energie, Stabilität oder Effizienz verbessern.

**Flow:**
1. Ausgangssystem laden  
2. Sandbox-Optimierung starten  
3. Varianten prüfen  
4. Constraints bewerten  
5. optimiertes System speichern  

---

## 11. User Story: Der Wissenspfleger

**Als Nutzer, der Ordnung schätzt**  
möchte ich sicherstellen, dass Steine konsistent, vollständig und korrekt sind  
um die Qualität des Graphen langfristig sicherzustellen.

**Flow:**
1. Liste aller unvollständigen Steine  
2. jeden öffnen  
3. fehlende Felder ergänzen  
4. Beziehungen validieren  
5. Stein veröffentlichen  

---

## 12. Zusammenfassung

Die User Stories zeigen, dass MUNDUS für viele Rollen nutzbar ist:
- Analysten  
- Erfinder  
- Ingenieure  
- Lernende  
- Forscher  
- Konstrukteure  
- Optimierer  
- Community-Mitwirkende  

Das System ist vielseitig, aber bleibt klar strukturiert und leicht verständlich.
